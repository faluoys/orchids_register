# 补全阶段人工接力设计

> 设计记录：为 “注册成功但需要登录后补全资料” 的 Orchids 账号增加桌面端人工接力流程。用户在应用内完成登录与真实访客验证，应用再继续刷新该账号的套餐与 credits。

## 背景

当前 Orchids 桌面端的账号主流程可以完成注册，并把 `created_session_id`、`created_user_id`、`client_cookie` 等信息落库。注册完成后，桌面端会异步调用现有 profile 刷新逻辑，尝试拉取账号的 `plan / credits`。

问题在于，部分账号在后续登录补全过程会触发网站的真实访客验证（当前观测为 `hCaptcha` “我是实际访客” 挑战）。这一步只能在真实浏览器交互中由用户手动完成。现状是用户需要自己跳到外部浏览器手工登录，应用没有明确的继续补全入口，也没有补全成功后的回收流程。

## 目标

- 对“注册成功但尚未补全”的账号，提供明确的“继续补全”入口。
- 在桌面应用内打开独立补全窗口，让用户手动完成登录和真实访客验证。
- 用户完成后，由桌面端主动检测该账号是否已能拉取 `plan / credits`，检测通过后更新账号列表。
- 全流程只覆盖补全阶段，不改动初始注册链路。

## 非目标

- 不自动识别、点击或绕过 `hCaptcha`。
- 不把初始注册阶段接入人工接力。
- 不实现多账号并行补全调度。
- 第一版不做补全窗口 cookie 直接导出回库。

## 方案对比

### 方案 1：系统浏览器打开登录页

优点：
- 实现快

缺点：
- 会话回收弱
- 用户体验割裂
- 很难和桌面端账号上下文绑定

### 方案 2：桌面端独立补全窗口

优点：
- 用户停留在应用内
- 可以绑定 `account_id`
- 可以在窗口关闭、检测、刷新之间做明确状态流转

缺点：
- 需要增加窗口管理和前后端通信

### 方案 3：外挂 Playwright/无头浏览器窗口

优点：
- 隔离性强

缺点：
- 成本高
- 不像桌面端内建流程
- 对首版过重

## 选型

采用方案 2：桌面端独立补全窗口。

## 用户流程

1. 用户在账号列表中看到一个账号已经注册成功，但 `plan / credits` 仍为空。
2. 用户点击“继续补全”。
3. 应用打开一个独立窗口，标题包含账号邮箱，页面加载 Orchids 登录/补全入口。
4. 用户在窗口内手动登录，并完成真实访客验证。
5. 用户点击“我已完成，继续检测”。
6. Tauri 后端对该账号执行一次“补全检测”：
   - 优先复用该账号已有 `created_session_id / created_user_id / desktop_jwt`
   - 调用现有 `fetch_plan_and_credits_with_session`
7. 若检测成功：
   - 更新数据库内该账号的 `plan / credits`
   - 通知主窗口刷新
   - 关闭补全窗口
8. 若检测失败：
   - 提示用户继续在窗口中操作
   - 不修改账号现有状态

## 状态设计

第一版不新增数据库字段，只在前端维护运行态：

- `idle`
- `opening`
- `waiting_user`
- `checking`
- `success`
- `failed`

原因：
- 这类状态是短时 UI 状态，不属于账号长期业务数据
- 数据库现有字段足以表达最终结果：`plan / credits` 是否已拿到

## 成功判定

不用“页面 URL 变化”或“窗口标题变化”这类弱信号。

唯一成功标准：

- 该账号在补全后可以成功跑通现有 profile 刷新链路，并拿到 `plan` 或 `credits`

这样可以直接对齐业务最终目标，避免补全过程看起来成功但账号仍不可用。

## 技术设计

### Tauri 后端

新增两类命令：

- `open_account_completion_window(account_id)`
  - 校验账号存在
  - 避免同一账号重复开窗
  - 创建新的 webview window
  - 窗口 URL 指向现有 Web 入口页，并附带 `accountId`

- `check_account_completion(account_id)`
  - 读取账号
  - 基于现有 `build_profile_session_context`
  - 调用 `fetch_plan_and_credits_with_session`
  - 成功时更新数据库，并返回最新账号
  - 失败时返回明确错误

另外增加：

- 简单的运行态管理，用于记录 “某个 account_id 当前是否已有补全窗口”
- 窗口关闭时清理内存态

### 前端

账号页新增：

- “继续补全”按钮
  - 仅对 `register_complete = true` 且 `plan / credits` 为空的账号展示

- 补全过程弹窗
  - 显示说明：需要手动登录并完成真实访客验证
  - 操作按钮：
    - 打开补全窗口
    - 我已完成，继续检测
    - 关闭

### 复用现有能力

优先复用：

- `src-tauri/src/commands/accounts.rs` 中现有账号读取和 profile 刷新逻辑
- `src-tauri/src/orchids_profile.rs` 中现有 `fetch_plan_and_credits_with_session`
- `ui/src/pages/AccountsPage.tsx` 中现有账号详情/刷新动作模式

## 错误处理

- 账号不存在：直接报错，不开窗
- 同一账号已存在补全窗口：聚焦已打开窗口，而不是重复创建
- 用户关闭窗口但未完成：保留账号原状态，不报成功
- 检测失败：给出“尚未检测到补全完成，请继续在窗口中操作”的提示

## 测试策略

### Rust

- 新增命令层测试：
  - account 不存在时返回错误
  - 重复开同一账号窗口时不会创建第二个运行态
  - 检测成功时会更新账号 plan / credits
  - 检测失败时不更新账号

### 前端

- 账号按钮显示条件测试
- 补全过程弹窗交互测试
- 检测成功后刷新当前账号状态

### 手工验证

1. 选中一个已注册但未补全的账号。
2. 点击“继续补全”。
3. 确认应用内打开独立窗口。
4. 在窗口内手动登录并完成真实访客验证。
5. 点击“我已完成，继续检测”。
6. 主列表显示 `plan / credits` 已刷新，补全过程结束。

## 风险

- 仅打开独立窗口并不能保证现有 session 一定可复用；部分账号可能仍需要后续补做“窗口会话回写”。
- 不同账号的补全入口页面可能有细微差异，第一版先使用统一登录入口，后续再按实际情况细化。

## 后续增强

- 检测成功后自动关闭窗口，不再要求用户手动收尾
- 补全窗口成功后的 cookie / session 回写
- 支持批量待补全账号的串行处理
